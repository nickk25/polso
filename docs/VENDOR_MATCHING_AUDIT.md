# Vendor Matching — Engineering Audit & Production Redesign

> **Implementation status (2026-06-13):** IMPLEMENTED in code, pending gated deploy. The redesign was built per the plan in `~/.claude/plans/proud-foraging-hinton.md` with a real test harness (`packages/banking/src/canonicalize.test.ts`, 68 passing) and validated on real production data (241 counterparties → 85 clean vendors, 0 wrongful merges). **Deliberate deviations from this audit, decided during implementation:** (1) TGSS is keyed at the agency level (`gov:tgss`, one vendor) rather than split by CCC — splitting would re-introduce fragmentation for an expense-intelligence product, and it removes the provisional-CCC-regex risk; (2) the IBAN hard-discriminator is realised as IBAN-first matching + a key-disambiguation suffix on same-name/different-IBAN collisions (the `@@unique` constraint forbids two identical-key rows); (3) `@polso/matching`'s `tokenize`/`normalizeName` were left intact (the receipt matcher depends on them) — the double-normalization fix lives in `merge-suggestions.ts` instead. **Still gated (not yet executed):** the DB schema migration, the `--apply` backfill, and the commit/push.

> **Original scope note:** the §4/§5 rules below were the *proposed* spec produced by an analysis workflow; §5's claimed pass-rate was generated by a model, not executed. The implementation treated them as a starting point and converged empirically against the real test harness + production dry-run (which surfaced and fixed several gaps the oracle missed, e.g. abbreviated `Tarj.`, `a favor de`, truncated city names).

## 1. Executive Summary

Polso's vendor matching is **fragmenting one real vendor into many counterparties** and, paradoxically, **suggesting merges of unrelated vendors**. Both failures trace to a single root cause.

**One-line root cause:** The deduplication key (`Counterparty.normalizedName`) is built from the raw bank-statement string with only payment-processor prefixes and legal suffixes stripped — it retains the Spanish operation verb (`compra`/`recibo`/`transferencia`), the literal `tarjeta` + 16-digit card PAN, `comision`, per-transaction reference codes, and city names. The key is therefore *per-transaction-unique* (causing fragmentation) while the surviving boilerplate tokens are *shared across all card purchases* (causing false-merge suggestions).

**Business impact (real production data, org `cmkhdqmia0000nuqgs6f6hyvw`):**

- **241 counterparties, 167 (69%) with junky normalized names.** Most "vendors" in the dashboard are transaction fragments, not entities.
- Analytics, recurring-expense detection, and per-vendor spend are all computed over fragmented counterparties, so vendor-level totals are wrong (Amazon spend is split across 16 rows, etc.).
- The merge-suggestion UI surfaces **false positives** (Apple + Zara), eroding user trust in the one tool meant to fix the mess.

**Fragmentation clusters (the disease, by count):**

| Cluster | Fragments | Nature |
|---|---:|---|
| transferencia | 44 | operation verb, not a vendor — should resolve to the person/IBAN |
| tgss | 23 | government — should key on agency + CCC |
| amazon | 16 | SaaS/retail — order codes (`Rv0u31hk4`, `Amzn Mktp`) |
| movil | 16 | generic operation — not a vendor |
| asesoria (Larrea) | 15 | receipt numbers in the key |
| canva | 11 | invoice IDs (`I04706-45793541`) |
| transaccion | 10 | generic operation — not a vendor |
| www / diputacion | 8 / 8 | generic / government (province = identity) |
| apple / devolucion / vercel / internet | 6 each | refs / generic / suffix / generic |
| lufthansa / facebk | 5 each | ticket numbers / ad codes |
| prime / railway | 4 each | order codes / TLD+ref |
| anthropic / zara | 3 each | `Pbc` suffix / locations |

**The verified false-merge (the symptom):** `compra apple com bill itunes com tarjeta 4176570` vs `compra zara donostia-san tarjeta 417657017008070`, both on the same corporate card, share `{tarjeta, <PAN>, comision, 00}` → **4 / 11 = 0.36 ≥ 0.3** Jaccard threshold → false merge suggestion despite zero brand overlap.

**Fix in one sentence:** Build a clean **vendor-identity matchKey at ingestion** (strip operation verbs, card/commission noise, refs, locations; carve out government entities by agency + discriminator; capture IBAN), make find-or-create *and* merge operate on that key, and demote name-similarity to a brand-blocked, human-confirmed **suggest** tier.

## 2. How Vendor Matching Works Today

End-to-end pipeline, with `file:line` references.

**Step 1 — Field selection (name source).** `transformTransactionName` (`packages/banking/src/transform.ts:23-39`) picks the counterparty name in priority order: `creditorName` (L24) / `debtorName` (L25) first, then `additionalInformation` (L26), `remittanceInformationStructured` (L27), `remittanceInformationUnstructured` (L30), `remittanceInformationUnstructuredArray` (L33). `transformCounterpartyName` (`transform.ts:57-61`) returns `debtorName`/`creditorName` or `null`.

- For card-POS transactions the issuer leaves `creditorName`/`debtorName` **empty**, so the name falls through to the bundled receipt string (`Compra Apple.com/bill … Tarjeta 4176570 …`). This is *why* 69% of keys are junk.
- `creditorAccount.iban` / `debtorAccount.iban` (`types.ts:200,202`) and `remittanceInformationStructuredArray` (`types.ts:187`) are **typed but never read** in `transformTransaction` (`transform.ts:74-120`).
- Ordering oddity: `additionalInformation` (L26) is checked **before** `remittanceInformationStructured` (L27) — rawer field wins if a bank sends both.

**Step 2 — Normalization (the dedup-key generator).** `transformTransaction` sets `name = description ?? transactionName`, `merchantName = counterpartyName ?? name` (`transform.ts:113`). At sync, `syncTransactionsCore` calls `normalizeCounterpartyName(tx.merchantName ?? tx.name)` (`apps/web/features/banking/lib/sync-core.ts:357-360`). `normalizeCounterpartyName` (`packages/banking/src/normalizer.ts:64-99`):

1. lowercase + collapse whitespace (L67)
2. strip `PREFIXES_TO_REMOVE` — **payment processors only** (PayPal, Square, Stripe, Google, Apple.com/bill, AMZN, Amazon), `^`-anchored (L69-72, list at L33-44). **No Spanish verbs.**
3. strip `SUFFIXES_TO_REMOVE` (S.L., S.A., LLC…), `$`-anchored (L74-77)
4. strip `#\d+`, `ref:\s*\d+`, `\*+\d+` (L82) — narrow; misses `Rv0u31hk4`, `I04706-45793541`
5. strip `\s+\d{4,}$` (L85) — `$`-anchored, so a trailing city/commission **after** the PAN defeats it
6. apply `VENDOR_MAPPINGS` (12 entries, L46-62) — **dead code**: every string starts with an unstripped `compra`/`recibo`, so `includes()` at L88 never matches
7. else strip `[^\w\s-]` and collapse (L93-96) — **dashes/locations retained**

`extractVendorFromDescription` (`normalizer.ts:104-123`) is exported but **never called**.

**Step 3 — Find-or-create + uniqueness.** Normalized name → `findOrCreateCounterparty` (`apps/web/features/counterparties/lib/counterparty-matcher.ts:18`):

- exact match on `normalizedName` via `@@unique([organizationId, normalizedName])` (`schema.prisma:313`; field at `:285`), L30-33
- else load **all** counterparties with `detectionPatterns` (L51-54) and bidirectional substring match `normalizedName.includes(pattern) || pattern.includes(normalizedName)` (L56-58) — **O(N²) per unmatched tx**
- else create, seeding `detectionPatterns` with the **full polluted** `normalizedName` (L68-78, seed at L75) — guaranteeing future variants never substring-match
- `type` upgrade to `"both"` (L37-41, L59-61) rarely fires because matches rarely succeed
- empty name currently **throws** (L20-22)

Stored as `Transaction.counterpartyName` (`sync-core.ts:400`).

**Step 4 — Merge suggestions (the symptom layer).** `computeMergeSuggestions` (`apps/web/features/counterparties/lib/merge-suggestions.ts:1-53`):

- `tokenize(normalizedName)` (L17) calls `normalizeName` from `@polso/matching` (`packages/matching/src/normalizers.ts:51-55`) — a **second normalization pass** that *does* strip one leading `compra`/`recibo` (L24-25) and replaces punctuation with space (L44) but **keeps numeric tokens** and only filters `length ≤ 1` (L54).
- greedy single-pass clustering (L24-40): anchor loops candidates, `jaccardSimilarity` (`matching/normalizers.ts:58-67`) ≥ `JACCARD_THRESHOLD = 0.3` (`merge-suggestions.ts:10,35-36`) → add to group, mark visited immediately (L38). Groups size ≥ 2 returned, sorted by entry count (L43).
- **Double normalization, no brand blocking, no transitivity check, order-dependent, no confidence.**

**Step 5 — UI / agent.** `counterparty-merge-suggestions.tsx` renders an amber banner (no confidence, no reject). `get-merge-suggestions.ts:1-19` exposes top-10 groups to the agent. One-click merge reassigns entries via `mergeCounterpartiesAction` (`actions/manage-counterparty.ts:213-275`).

## 3. Root-Cause Analysis (ranked)

The decisive distinction: **(A) an ingestion fragmentation disease** (the dedup key is per-transaction-unique) and **(B) a merge-suggestion symptom** (similarity runs over polluted tokens). (B) is downstream of (A); fixing the suggestion threshold alone treats the symptom.

**R1 — DISEASE (root). The dedup key embeds per-transaction tokens.** `normalizeCounterpartyName` (`normalizer.ts:64-99`) never strips the Spanish operation verb, `tarjeta <PAN>`, `comision`, embedded refs, or city names. Every Amazon order code (`Rv0u31hk4` vs `Na58c3tm4`) yields a distinct `normalizedName`, so `@@unique([organizationId, normalizedName])` (`schema.prisma:313`) creates a **new row per transaction**. Directly produces amazon 16, canva 11, asesoria 15, vercel 6, etc. *Examples:* `compra amazon rv0u31hk4 luxembourg tarjeta 41…` ≠ `compra www amazon na58c3tm4 luxembourg tarjeta 41…`.

**R2 — DISEASE. The name source is the wrong field for card-POS rows.** Card issuers leave `creditorName`/`debtorName` empty (`transform.ts:24-25,57-61`), forcing the name to fall through to the unstructured receipt string. The single highest-value stable identifier — `creditorAccount.iban`/`debtorAccount.iban` (`types.ts:200,202`) — is **dropped entirely** in `transformTransaction` (`transform.ts:74-120`). This is why the 44-row `transferencia` cluster fragments even though a stable beneficiary IBAN exists in the payload.

**R3 — DISEASE. Operation verbs and government bodies are treated as vendors.** `transferencia` (44) and `transaccion` (10) are bank operations, not vendors; `tgss` (23) and `diputacion` (8) are government entities whose identity is **agency + discriminator** (CCC for TGSS, province for Diputación), not the bare agency name. The current pipeline has no carve-out, so it fragments these on the surviving refs and would (under any naive cleanup) collapse distinct obligors/provinces.

**R4 — SYMPTOM (the verified bug). Jaccard over polluted tokens + 0.3 threshold + no brand key.** Because R1 leaves `{tarjeta, <PAN>, comision, 00}` in the key, two unrelated vendors on the same card share those tokens. **Apple vs Zara: intersection 4, union 11, Jaccard 0.36 ≥ 0.3 → false merge** (`merge-suggestions.ts:35-36`). The numeric PAN (`4176570170080`) is the dominant overlap driver and is filtered by neither normalizer. There is no minimum-brand-token requirement, no numeric exclusion, no CIF match — similarity can pass on 90% boilerplate + 0% brand.

**R5 — SYMPTOM. Double normalization + greedy clustering.** `tokenize` re-normalizes an already-normalized key (`matching/normalizers.ts:51-55`), so the two normalizers disagree on what `compra` means. Greedy single-pass clustering (`merge-suggestions.ts:24-40`) marks candidates visited on first match — order-dependent, no transitivity validation, no cohesion metric, no confidence score, no reject path.

**R6 — DISEASE (latent). `detectionPatterns` seeded with the full polluted key.** `counterparty-matcher.ts:75` guarantees substring matching can never rescue a variant, and the O(N²) scan (L51-54) degrades as counterparties grow.

## 4. Redesign

No code is changed in this run. This is the rule set an implementer follows.

### 4.1 Ingestion — prefer structured fields, capture IBAN

Replace `transformCounterpartyName` field priority (`transform.ts:57-61`) with:

1. `creditorName` (money out) — structured, trust fully → set `nameIsStructured = true`
2. `debtorName` (money in) — structured → `nameIsStructured = true`
3. `remittanceInformationStructured` (**move before** `additionalInformation`)
4. `remittanceInformationStructuredArray[0]` (typed at `types.ts:187`, wire in)
5. `additionalInformation`
6. `remittanceInformationUnstructured`
7. `remittanceInformationUnstructuredArray[0]`
8. `proprietaryBankTransactionCode` → else `"Unknown"`

When `nameIsStructured`, the canonicalizer **skips** the aggressive digit/boilerplate strip (a structured payee is already clean; the digit-run rule could eat a legit numeric brand).

**IBAN capture (additive):** extract `creditorAccount?.iban ?? debtorAccount?.iban` in `transformTransaction`, carry on `BankTransaction`, store as new nullable `Transaction.counterpartyIban`.

### 4.2 Canonical normalizer — `canonicalize(name, { structured }) → { matchKey, displayName, seenLocations }`

Run **once at ingestion** (never at match time). Idempotent (required for migration). Two decoupled outputs:

- **`matchKey`** — identity for dedup/auto-merge. Sorted lowercase token set (vendors) or `gov:<agency>:<discriminator>` (government). Never reshaped by display cosmetics.
- **`displayName`** — human label, order-preserving, original casing. Cosmetic only.

**§4.2.0 — Government carve-out (runs FIRST, before any stripping).** This resolves the two adversarial blockers (Diputación and TGSS) — the discriminating tokens are captured *before* location-strip (§4.2.4) and the digit-killer (§4.2.2) can destroy them. Tight detection set so it cannot catch a normal vendor:

```
GOV_BODIES (first match wins):
  /diputaci[oó]n\s+foral\s+de\s+(\w+)/                         → key gov:diputacion foral de:<province>
  /\btgss\b/ or /tesorería general de la seguridad social/      → key gov:tgss:<ccc>
  /\bayuntamiento\s+de\s+(\w+)/                                → key gov:ayuntamiento de:<muni>
  /agencia tributaria|\baeat\b|hacienda\s+(foral|estatal|pública|tributaria)/ → key gov:agencia tributaria
```

- **Province is the discriminator**, NOT a location: `gov:diputacion foral de:gipuzkoa` ≠ `gov:diputacion foral de:bizkaia` — never auto-merged. This is why adding `bizkaia`/`araba` to the LOCATIONS list (§4.2.4) can never collapse provincial authorities: gov rows never reach §4.2.4.
- **CCC** (`régimen + account`, e.g. `28 0123456789`) extracted via `/\b(\d{2})\s+(\d{8,12})\b/` *before* the digit-killer; `Periodo` digits are **not** captured → same-CCC/diff-period merge, diff-CCC stay distinct: `gov:tgss:280123456789` ≠ `gov:tgss:480987654321`.
- **⚠️ The CCC regex is derived from two synthetic samples** — validate against real GoCardless TGSS descriptors (régimen-code width, separators, presence of `Periodo`) before shipping. Treat as provisional.
- `hacienda` is qualified (`hacienda foral/estatal/…`) so "Hacienda Los Olivos" (a restaurant) routes to the vendor branch (`hacienda olivos`), not gov.

**§4.2.1 — Pre-token rewrites (vendor branch; skipped for structured names):**

```
PROCESSOR_REWRITES (order matters):
  /\bamzn\s+mktp\b[^,]*/   → " amazon "   // RESTORED from matching/normalizers.ts:5 (v1 had dropped it)
  /\bamzn\s*\*\s*/         → " amazon "
  /\bwww\.amazon\b[^,\s]*/ → " amazon "
  /\bamazon\s*\*\s*/       → " amazon "
  /\bapple\.com\/bill\b/   → " apple "
  /\b(paypal|pp|sq|square|sp|stripe|google)\s*\*\s*/ → " "   // processor passthrough reveals real vendor
Card/commission noise:
  /\btarjeta\b[\s\d]*/     → " "   // drops "Tarjeta 4176570…" incl. trailing PAN
  /\bcomision\b[\s\d.,]*/  → " "   // drops "Comision 0,00"
Memo tail (fixes Asesoria "bbt" + transfer memos):
  /\b(concepto|nº\s*recibo|ref(erencia)?)\b.*$/ → " "
Punctuation: /[^\w\s]/ → " " ; /_/ → " "
```

**§4.2.2 — Tokenize + reference/PAN killer:**

```
DROP token if:
  /^\d+$/.test(t)                       // pure numeric (PAN fragments, "00")
  || /\d{2,}/.test(t)                   // any run of 2+ digits
  || (/\d/.test(t) && /[a-z]/.test(t))  // alnum mix: rv0u31hk4, i04706, ex6gk5, rf5
UNLESS t ∈ BRAND_ALLOWLIST (3m, m6, o2, s4 — checked first; grows via junk-token log §6.5)
```

**§4.2.3 — Location → metadata (vendor rows ONLY; gov rows already exited at §4.2.0):**

```
seenLocations = tokens ∩ LOCATIONS   // curated explicit list, never heuristic
tokens        = tokens − LOCATIONS
```
A vendor in multiple cities is ONE vendor (binding decision): Zara Donostia + Madrid → `zara`, `seenLocations` accumulates. The province in a gov name was already consumed at §4.2.0, so the §4↔§5 contradiction the adversary flagged does not exist in this design — gov rows never run location-strip.

**§4.2.4 — Subsidiary / suffix / stopword strip:**

```
SUBSIDIARY_WORDS = { iberica, espana, spain, europe, europa, international }   // fixes IKEA Iberica; NOT france (brand-essential)
LEGAL_SUFFIX     = { sl,slu,sa,sau,sc,sll,scp,inc,llc,ltd,limited,corp,gmbh,ag,bv,nv,
                     pbc,oy,oyj,sas,sarl,spa,srl,plc,kg,kft,sociedad,limitada,co,company }  // pbc fixes Anthropic
STOPWORDS        = { compra,recibo,pago,cargo,abono,adeudo,devolucion,transferencia,transaccion,
                     bizum,trf,transf,tarjeta,comision,concepto,de,del,la,el,los,las,a,en,
                     web,www,com,es,co,the,help,trip,bill,movil,internet,online }
tokens = tokens.filter(t => not in any set, length > 1)
tokens = dedupe(preserve first-seen order)
```
`net`/`org`/`app` are **deliberately NOT stopwords** — legitimate brand words ("The Net Lab", "Railway.app"); the `.net`/`.app` TLD attaches to a domain token handled by the punctuation split + digit/ref killer, not blanket removal. (See §7 residual on "The Net Lab".)

**§4.2.5 — Alias map + key/display assembly:**

```
ALIASES (first match on joined token string wins) — restored production VENDOR_MAPPINGS folded in:
  apple, prime video|primevideo→"prime video", amazon, facebk|facebook→facebook, netflix, spotify,
  vercel, railway, anthropic, canva, wix, uber, cabify, lufthansa, air france,
  ikea, "zara home" (before zara), zara, bershka,
  // restored from normalizer.ts:46-62 (NOT in the oracle, so carried explicitly to avoid silent regression):
  iberdrola, endesa, naturgy, vodafone, movistar, orange, mercadona, carrefour, lidl
  // DECISION (retained from normalizer.ts:51): "amazon prime"/"prime video" stays DISTINCT from "amazon"

matchKey    = alias ? sort(alias tokens).join(" ") : sort(unique surviving tokens).join(" ")
displayName = alias.display ?? join(surviving tokens in original order/casing, keeping a leading "The")
```

### 4.3 Brand/identity key — one key for find-or-create AND merge

- `Counterparty.normalizedName` = **matchKey** (vendor sorted-token set or `gov:…`); `@@unique([organizationId, normalizedName])` now enforces entity-level dedup.
- `Counterparty.name` = **displayName**.
- `detectionPatterns` seeded with **clean brand tokens**, never the polluted string (fixes R6).
- `tokenize()` in `@polso/matching` reduced to a **plain split** of the clean `normalizedName`; **delete the second `normalizeName()` pass** (`matching/normalizers.ts:51-55`) — double-normalization gone (fixes R5).
- **Match order in `findOrCreateCounterparty`:**
  1. **gov rows:** `gov:` matchKey exact → match/create. IBAN **not** used to merge gov rows (shared collector IBAN risk, §4.1/§7).
  2. **non-gov, IBAN present:** stored `Counterparty.iban` exact → match (unifies transferencia 44).
  3. **matchKey exact** (indexed) → match.
  4. else **create.**
- Drop the O(N²) substring scan over `detectionPatterns` (`counterparty-matcher.ts:51-65`).
- **Empty matchKey → no-op return, create NO counterparty** (change the current throw at `counterparty-matcher.ts:20-22`); store the transaction with `counterpartyName = null`. Suppresses generic rows (`movil` 16, `transaccion` 10, `www` 8, `internet` 6).

### 4.4 Similarity / merge — two tiers

Empirically (clean tokens) `asesoria larrea` vs `asesoria larrea bbt` and `zara` vs `zara home` are token-structurally identical (each a one-token superset). **No threshold separates them → similarity must never auto-merge.**

**Tier 1 — AUTO (deterministic, at ingestion):**
- non-gov: IBAN exact OR byte-identical `matchKey` OR exact CIF/NIF.
- gov: byte-identical `gov:` matchKey only. Gipuzkoa ≠ Bizkaia and CCC-28 ≠ CCC-48 therefore **never auto-merge** — the discriminator is in the key.
- Apple ≠ Zara falls out for free (disjoint tokens; PAN/`tarjeta` stripped at key time).

**Tier 2 — SUGGEST (similarity, never auto-applied):**
- Inputs are clean brand tokens (Apple+Zara collision structurally impossible).
- **Brand-anchored blocking:** candidate only if it shares ≥ 1 brand token (Apple/Zara share zero → never scored).
- Jaccard ≥ **0.5** → suggestion, confidence `(jaccard − 0.5)/0.5`.
- **No greedy transitive over-merge:** build bidirectional pairs; group A-B-C only if A-C also clears the bar.
- Resolutions: `zara`↔`zara home` (shared `zara`, 0.50, low conf → user **rejects**); residual ref/domain variants → high conf → user merges.

**UI / agent:** show confidence, gray out low-confidence (`counterparty-merge-suggestions.tsx`); add a **persisted "Don't merge"** dismissal so Zara/Zara-Home does not resurface; `get-merge-suggestions.ts` returns confidence and requires user approval for low-confidence groups.

### 5.4 Documented deviation (1/28)

The oracle expects display `The Net Lab`; the pipeline yields `The Net Lab Thenet` because the domain `thenet-lab.co` tokenizes to a `thenet` token that cannot be reconciled with the spelled-out name **without a cosmetic rule that would reshape the matchKey and cause fragmentation** (full form keys `lab net thenet`; domain-only form keys `lab thenet` — same vendor, two keys). If exact display is later required, apply domain-echo stripping to *display tokens only*, never the key.

## 6. Migration & Backfill Plan

Executable, reversible, idempotent. Dry-run first.

### 6.0 Prerequisite refactors (resolve two adversarial majors)

- **Extract `mergeCounterpartiesCore(tx, { organizationId, sourceIds, targetId })`** from `mergeCounterpartiesAction` (`actions/manage-counterparty.ts:213-275`). The action's first line `getAuthContext()` (≈L217) reads the request session and **throws in a node/tsx script** — so the "reuse the action" plan is not executable. The Server Action becomes a thin wrapper: `getAuthContext()` → `prisma.$transaction(tx => mergeCounterpartiesCore(tx, {organizationId, …}))`. The migration imports `mergeCounterpartiesCore` directly and passes the org id explicitly.
- **Fix the `DismissedPattern` orphan** (pre-existing bug the migration would multiply). `Counterparty.dismissedPatterns` (`schema.prisma:311`) has `onDelete: SetNull` (`schema:538`). The action reassigns `Entry` (≈L245) and `RecurringPattern` (≈L250) but **not `DismissedPattern`** — so `deleteMany` on sources nulls `DismissedPattern.counterpartyId`, and `detect-patterns-core.ts:57-60` silently resurfaces dismissed patterns on the survivor. Add, before `deleteMany`:
  ```
  tx.dismissedPattern.updateMany({ where: { counterpartyId: { in: sourceIds } }, data: { counterpartyId: targetId } })
  ```

### 6.1 Steps (idempotent; dry-run first)

1. **Compute** `{matchKey, displayName, seenLocations}` for every existing `Counterparty` by running `canonicalize()` on its `name` (raw display). Where `name` is itself dirty, recompute from the most-frequent `Transaction.counterpartyName`/`merchantName` of linked entries.
2. **Group** by new `matchKey`. Size ≥ 2 = a collision (fragments of one entity).
3. **Persist a durable dry-run report** *before any write* (resolves the irreversibility major): write each collision group to a `counterparty_merge_audit` table (or a JSON artifact in R2 via `@polso/storage`) — proposed survivor (max `_count.entries`, reusing the sort at `merge-suggestions.ts:43`), entries to reassign, and rows whose new matchKey is empty (generic → flag). **Default = report only, no writes.** This is the review artifact AND the reverse map.
4. **Apply (explicit flag):** per collision group call `mergeCounterpartiesCore` inside `prisma.$transaction`. Collapse fragments **before** any rename → no unique-constraint violation.
5. **Rename survivors:** set `normalizedName = matchKey`, `name = displayName`; **reseed `detectionPatterns` with clean tokens**; populate `seenLocations` and `iban`; record the `mergedFrom` audit field. Renames cannot collide — one survivor per group.
6. **Generic rows** (empty matchKey): entry-less → delete; with entries → detach (`counterpartyId = null`) then delete the junk row.
7. **Backfill transactions:** recompute `Transaction.counterpartyName = matchKey`, populate `Transaction.counterpartyIban` from re-fetched payloads (or leave null until next sync). Relink orphaned entries against canonical keys — **verify the relink path covers income/`both`-side entries**, not only expense (`backfill-counterparties.ts:18-37` filters `direction='expense'`; see §7 residual).

**Reversibility (partial — read this):** `Counterparty.mergedFrom Json?` records each absorbed source's `{id, name, normalizedName}`, so the vendor *rows* can be recreated. It does **not** record which entries belonged to which source — the merge reassigns all source entries onto the survivor and deletes the sources, so a clean re-split is not possible. **The real safety gate is the dry-run report, not undo.** Review `vendor-backfill-report.json` before running `--apply`.

**Idempotency:** `canonicalize` is idempotent and merges target the deterministic max-entry survivor → a second run finds zero collisions, zero renames — *for vendor/expense entries* (income-side caveat in §7).

**Ordering:** refactor (6.0) → dry-run + persist → human review → merge-collisions (`mergeCounterpartiesCore`, incl. `DismissedPattern`) → rename + reseed survivors → handle generics → backfill → re-run to confirm no-op.

### 6.2 Deploy sequence — FORCED ORDER (do not reorder)

The new code assumes `Counterparty.normalizedName` is already the clean matchKey. Un-migrated rows still hold the polluted string, and `merge-suggestions.ts` now plain-splits the key (stopword filtering moved into `canonicalize`), so on un-migrated data its behaviour is fragile (Apple+Zara sits just under the 0.5 threshold by coincidence) and new clean-keyed syncs won't dedupe against old polluted rows. Therefore:

1. **Apply the schema migration** (`db:migrate`/`db:push`) — adds `Counterparty.iban/seenLocations/mergedFrom`, `Transaction.counterpartyIban`. (Code already typechecks against the regenerated client.)
2. **Run the backfill `--apply`** (`apps/web/scripts/backfill-vendor-identity.ts`, against the prod `DATABASE_URL` via `tsx` — needs step 1, not the Vercel deploy). **Review the dry-run report first.**
3. **Commit + push** (the per-layer commits) to deploy the new code.

Pushing before step 2 ships a broken intermediate state (still-duplicated vendors, fragile suggestions). Run the backfill once more without `--apply` afterwards to confirm idempotency (expect 0 collisions). Similarity stays strictly **suggest**-tier — never auto-apply a Jaccard result.

### 6.3 Observability

- Log per-transaction `% source tokens dropped` (digit-run + stopword) — spike = new bank format.
- Log `matchKey === ""` suppression count — rise = a real vendor being eaten.
- Log AUTO-merge counts by reason (gov-key / IBAN / matchKey / CIF) and SUGGEST accept/reject rate — high rejection of *high*-confidence suggestions → recalibrate the 0.5 threshold.
- Log gov-branch hit rate by agency — a vendor wrongly entering the gov branch shows as an unexpected `gov:` key.

## 7. Risks & Open Questions

**Adversarial blockers — RESOLVED in this design (state confirmed for the implementer):**

- **Diputación Gipuzkoa vs Bizkaia auto-merge (blocker).** Resolved by the §4.2.0 government carve-out running *before* location-strip; province is the discriminator in `gov:diputacion foral de:<province>`. The §4↔§5 contradiction the adversary flagged does not exist because gov rows never reach §4.2.4. Adding provinces to LOCATIONS is now harmless.
- **TGSS CCC-28 vs CCC-48 auto-merge (blocker).** Resolved by extracting the CCC *before* the digit-killer; key `gov:tgss:<ccc>`. Period digits still dropped (same-CCC/diff-period merge correctly).
- **AMZN MKTP / Anthropic PBC / IKEA Iberica fragments (majors).** Resolved by restoring the `^amzn mktp` rewrite, generalizing `LEGAL_SUFFIX` (pbc, oy, sas, sarl, …), and adding `SUBSIDIARY_WORDS`.
- **Migration can't call a Server Action; DismissedPattern orphan; irreversibility (majors).** Resolved by §6.0 extraction, the `DismissedPattern` reassign, and the persisted audit report + `mergedFrom`.

**Residual / open items (NOT fully closed — flagged honestly):**

1. **CCC regex is provisional.** Derived from two synthetic samples. **Must be validated against real GoCardless TGSS descriptors before ship** (régimen-code width, separators, presence of `Periodo`). If the real format differs, the TGSS branch is wrong.
2. **"The Net Lab" display deviation (1/28).** Display renders `The Net Lab Thenet`; the two renderings collapse at SUGGEST, not AUTO. Accepted residual — the alternative (echo-strip into the key) was rejected because it reshapes the key cosmetically and would re-fragment.
3. **Idempotency overstated for income/`both` side.** `backfill-counterparties.ts:18-37` relinks only `direction='expense'` orphans; income-side entries detached in step 6/7 are never relinked, so a re-run is not a *clean* no-op for client/`both` counterparties. **Verify and broaden the relink path before claiming the re-run is a no-op.**
4. **Government IBAN ambiguity.** SEPA-debit TGSS/AEAT receipts may share the *collector's* IBAN across distinct obligors, so IBAN-first could reinforce a wrong gov merge — that is precisely why §4.3 uses gov-key ahead of IBAN for government rows. Confirm no gov row falls through to the IBAN branch.
5. **Self-inflicted regression watch.** The restored production `VENDOR_MAPPINGS` (iberdrola, endesa, naturgy, vodafone, movistar, orange, mercadona, carrefour, lidl, spotify) are **not in the oracle**, so the harness cannot catch their loss — carried explicitly; do not drop them.
6. **Over/under-strip escape hatches.** `BRAND_ALLOWLIST` (3M/M6/O2/S4) guards the digit-run rule from eating alnum brands; LOCATIONS is a curated explicit list, never heuristic. Both grow via the junk-token log.

**Judgment calls (face-value, documented — change only with product sign-off):**

- **Zara vs Zara Home** → distinct sub-brands → SUGGEST (rejectable), never AUTO.
- **Zara vs Bershka** → distinct Inditex brands.
- **Prime Video / Amazon Prime vs Amazon marketplace** → distinct (consistent with the existing `normalizer.ts:51` decision).
- **Bizum-to-person vs Transferencia-to-same-person** → kept NO-MERGE to avoid over-aggressive partial-name person matching; flagged as an edge case rather than asserting a merge.
